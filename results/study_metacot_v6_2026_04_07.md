## 1. Executive Summary

### 1.1 Goal

Train a Meta-CoT model (Qwen3-8B) that exceeds the base SFT baseline (67.1% on 1,030 mixed math problems) through metacognitive control -- specifically, by enabling three key behaviors at inference time:

1. **Route switching**: when the current solution approach fails, the model diagnoses the failure and pivots to a structurally different method.
2. **Overconfidence correction**: the model maintains calibrated confidence that tracks actual correctness, reducing false-positive rate on hard problems.
3. **Stepwise reasoning with verification**: the model performs independent verification of its own answers, catching and correcting errors before committing to a final answer.

These behaviors are mediated by `<|meta|>` token blocks interleaved with ordinary chain-of-thought reasoning. The meta blocks externalize five metacognitive states: local confidence, anomaly detection, failure diagnosis, control action (verify or redirect), and study needs.

### 1.2 Current Status

| Phase | Status | Key Result |
|---|---|---|
| Phase 0: V5 analysis | Complete | 7 findings, diagnosis-execution gap identified |
| Phase 1: SlotC gate | Complete | accuracy 67.3% (above base), approach_change 0.3% (gate failed) |
| Phase 2: V6 SFT (clean data) | In progress | E19/E19b/E19c launched, eval pending |
| Phase 3: V6.2 RL | Gated | Requires E19 approach_change >= 5% and accuracy >= 67.1% |

### 1.3 One-Sentence Summary

V5 demonstrated that Meta-CoT produces perfect diagnoses but zero route switching; V6 traces this to theatrical redirect data (86% from gpt-5.4-mini) and insufficient RL steps (200 steps = 0.4 epochs), motivating a clean-data restart with gpt-5.4 full redirects and a 1000+ step RL plan.

## 2. Key Discoveries

### 2.1 The Theatrical Redirect Problem

**Intent (의도)**: Understand why E11 achieved approach_change of approximately 0%, despite SFT on redirect exemplars.

**Hypothesis (가설)**: The SFT training data itself may lack genuine route-switching behavior -- the model cannot learn to switch if the exemplars do not demonstrate switching.

**Verification (검증)**:

Analysis of the V5 redirect data (generated by gpt-5.4-mini) revealed that 86% of redirect chains were "theatrical": they had pre-meta length of 0 characters. This means the model was trained on examples where the `<|meta|>` block appeared at the very beginning, before any actual problem-solving attempt. The model learned the pattern "diagnose then solve directly" rather than the intended pattern "try, fail, diagnose, switch, succeed."

In contrast, gpt-5.4 full (model `gpt-5.4_2026-03-05`) produces 0% theatrical redirects and 80% natural redirects, with a mean pre-meta length of 378 characters. This was confirmed in a smoke test of 25 chains:

| Source | Theatrical (pre=0) | Natural (pre>=200) | Mean Pre-Meta Length |
|---|---|---|---|
| gpt-5.4-mini | 86% | 14% | ~40 chars |
| gpt-5.4 full | 0% | 80% | 378 chars |

**Conclusion**: The theatrical redirect problem was the root cause of E11's approach_change = 0%. The model was never exposed to the "try, fail, switch" pattern during training. This discovery motivated the complete V6 clean-data restart using gpt-5.4 full for redirect generation.

### 2.2 RL Step Insufficiency

**Intent (의도)**: Determine whether previous RL experiments ran long enough to learn behavioral rewards.

**Hypothesis (가설)**: The prior RL experiments used too few training steps relative to the data size, preventing convergence of behavioral reward signals.

**Verification (검증)**:

Previous GRPO runs (E8-E10) used only 200 steps. With an effective batch size of 16 on approximately 8K math problems, one epoch requires approximately 500 steps. Thus 200 steps corresponds to 0.4 epochs -- less than a single pass through the data. For reference, the behavior-uncertainty literature (PPO on 100K data) reports requiring 5,859 steps (15 epochs) for behavioral reward convergence.

| Metric | Previous RL | Behavior-Uncertainty Reference | V6 Plan |
|---|---|---|---|
| Training steps | 200 | 5,859 | 1,000-1,500 |
| Epochs | ~0.4 | ~15 | ~2-3 |
| Data size | ~8K | ~100K | ~6.3K |

**Conclusion**: 200 steps is fundamentally insufficient for behavioral reward learning. The V6 plan mandates a minimum of 700 steps (~1.5 epochs), with the mainline target at 1,000 steps (~2 epochs).

### 2.3 Reward Evolution: Four Generations

The project has iterated through four generations of reward design, each addressing failures observed in the previous generation.

**1st Generation (E1-E4): Basic additive rewards.**

| Mode | Rewards | Weights | Issue |
|---|---|---|---|
| E1 | correctness + format | 1.0, 0.5 | No metacognitive signal |
| E2 | + meta_quality | + 1.0 | Meta quantity, not quality |
| E3 | + calibration | + 0.5 | 100% meta coverage, lowest accuracy (51.7%) |
| E4 | + uncertainty_meta | + 0.5 | Too many rewards, signal collapse |

The core problem: adding more reward components to a single sum caused the correctness signal to be drowned by metacognitive reward noise.

**2nd Generation (E5-E10): Behavioral decomposition.**

| Mode | Rewards | Key Change |
|---|---|---|
| E5 | E3 + confidence_revision | Revision-only test |
| E6 | E3 + probe_calibration | Probe-based calibration (collapsed) |
| E7 | E6 + stepwise_probe | Probe + blockwise scoring |
| E8 | E5 + overconfidence_penalty + length | Strongest calibration (ECE 0.129) |
| E9 | E8 + verify + overconfidence_verify | Best RL overall (62.1%) |
| E9b | E8 + redirect | Redirect-only test |
| E9c | E8 + diagnosis + decomposition | Diagnosis-only test |
| E10 | E8 + all behaviors | Full controller, worst accuracy (53.9%) |

Key findings: E9 achieved the best RL accuracy (62.1%) with verify-focused rewards. E10 demonstrated that combining all rewards degrades accuracy. E6 showed probe calibration collapse (reward converged to 0.0 via trivial agreement). The decomposition (E9/E9b/E9c) was valuable for isolating individual behavior effects.

**3rd Generation (E9v2-E10v2): Targeted fixes.**

| Mode | Fix Applied |
|---|---|
| E9v2 | + same-route repetition penalty + confidence omission floor |
| E9bv2 | + route_switch_evidence + confidence omission floor |
| E10v2 | All V2 fixes combined |

These addressed specific reward-hacking behaviors (verify by restating the same calculation, escaping calibration by suppressing meta emission) but still carried too many reward components (up to 17 in E10v2).

**4th Generation (E12-E13): Simplified structural rewards.**

| Mode | Rewards | Weights | Design Principle |
|---|---|---|---|
| E12 | correctness + structural_switch | 1.0, 0.3 | 2 rewards only: switch-focused |
| E13 | correctness + structural_switch + confidence_trajectory + verify_outcome | 1.0, 0.3, 0.3, 0.2 | 4 rewards: full V6.2 design |

The 4th generation eliminates all proxy rewards (format, meta_quality, length_penalty) and retains only causally meaningful signals. Each reward directly measures a behavior linked to correctness improvement.

### 2.4 GDPO vs GRPO

**Intent (의도)**: Preserve individual reward signals when combining multiple reward functions.

**Hypothesis (가설)**: Standard GRPO (sum rewards, then normalize per group) collapses distinct reward contributions into a single scalar, making it impossible to attribute advantage to specific behaviors. GDPO (normalize each reward independently, then weight-sum) preserves individual signal fidelity.

**Verification (검증)**:

The GDPO monkey-patch in `src/training/grpo_v2.py` implements per-reward group normalization:

```
GRPO: sum(rewards) -> group_normalize           (collapses distinct reward combos)
GDPO: group_normalize(each_reward) -> sum -> batch_normalize  (preserves signal)
```

Reference: arXiv:2601.05242 (NVIDIA NVlabs). GDPO is activated for all multi-reward modes (E3-E13). All RL experiments in this project use GDPO advantage computation.

The practical impact: without GDPO, a strong correctness reward (+3.0 weight) would dominate, and a weaker structural_switch reward (+0.3 weight) would contribute negligible gradient. With GDPO, both are group-normalized before weighting, so the switch reward retains its relative influence within each generation group.

### 2.5 SlotC Gate Result

**Intent (의도)**: Determine whether stronger SFT seeding on the E9 checkpoint can break the approach_change = 0% bottleneck.

**Hypothesis (가설)**: If 164 curated redirect exemplars are trained for 5 epochs (instead of 2) on top of the E9 checkpoint, structural switching behavior will emerge.

**Verification (검증)**:

SlotC (E9 + 164 seed x 5 epochs SFT) was evaluated on the standard 1,030-problem benchmark:

| Metric | Value | Gate Threshold | Pass? |
|---|---|---|---|
| Accuracy | 67.3% | >= 60% | Yes |
| Approach change | 0.3% | >= 5% | **No** |
| ECE | 0.109 | -- | Improved |
| Multi-meta | 49.9% | -- | Strong |
| False-positive rate | 25.6% | -- | Improved from 30.0% |
| Confidence coverage | 98.2% | -- | Near-complete |

**Conclusion**: The E9-line SFT can improve calibration (ECE 0.109, FP 25.6%) and information checkpointing (multi-meta 49.9%) but cannot create structural switching behavior. The 5-epoch seeding with curated redirects failed to break the approach_change bottleneck. This confirms that the E9 checkpoint's habits are too deeply entrenched for post-hoc seeding to overcome.

**Decision**: Case B of the decision tree is triggered. The E9-based seed SFT line is no longer mainline. A clean-data restart from base_sft with V6 10K data becomes the active mainline.

## 5. RL Plan: Phase 3

### 5.1 V6.2 Reward Design

Phase 3 RL uses the 4th-generation reward design (E12/E13). Each reward function is detailed below with its intent, hypothesis, and verification mechanism.

#### 5.1.1 R1: correctness_reward (weight 1.0)

**Intent (의도)**: Ensure the model's primary optimization target remains producing correct mathematical answers.

**Implementation**: Sympy-based math verification (Open-R1 style). Handles fractions, decimals, LaTeX, boxed answers with fallback to string matching.

**Weight rationale**: Correctness is the dominant reward at weight 1.0. All other rewards are scaled below 0.5 to ensure correctness is never dominated.

#### 5.1.2 R2: structural_switch_reward (weight 0.3)

**Intent (의도)**: Reward genuine method changes -- not just meta text about switching, but actual structural differences in the problem-solving approach before and after the meta block.

**Hypothesis (가설)**: If the model receives positive reward only when the method family changes across the meta boundary AND the final answer is correct, it will learn to switch methods when needed rather than producing theatrical redirects.

**Implementation** (`rewards.py`, line 1376):

1. Locate `<|meta|>` and `<|/meta|>` boundaries.
2. Extract pre-meta text (before `<|meta|>`) and post-meta text (after `<|/meta|>`).
3. Require both regions to be >= 30 characters (preventing theatrical patterns).
4. Check `_methods_structurally_differ(pre, post)`: uses 10 method-pair heuristics (e.g., algebra/invariant, direct/case-split, coordinate/vector, counting/generating-function) plus explicit switch declarations.
5. Award +1.0 only if methods differ AND the answer is correct. Otherwise 0.0.

**Verification (검증)**: Success is measured by approach_change rate on the evaluation benchmark. Target: >= 5%.

#### 5.1.3 R3v2: confidence_trajectory_reward (weight 0.3)

**Intent (의도)**: Shape the confidence trajectory across multiple meta blocks to reflect healthy metacognition: calibrated final confidence, gradual (not abrupt) changes, and directionally appropriate movement (rising for correct answers, falling for incorrect).

**Hypothesis (가설)**: A 3-axis continuous scoring system will encourage multi-meta emission and discourage both over-confident wrong answers and under-confident correct answers, without the collapse risks seen in probe-based calibration (E6).

**Implementation** (`rewards.py`, line 1426):

For completions with multiple confidence values:

- **Axis 1: Calibration (weight 0.4)**: Brier score on last confidence vs. actual correctness. `cal = 1.0 - (last_conf - correct)^2`.
- **Axis 2: Gradual change (weight 0.3)**: Penalizes abrupt confidence jumps. `gradual = max(0, 1.0 - max_jump * 2)`.
- **Axis 3: Direction (weight 0.3)**: Correct answers should show rising confidence; wrong answers should show falling confidence. `dir = max(0, min(direction * 2, 1.0))` or `min(-direction * 2, 1.0)`.

Special cases:
- Single confidence: discounted Brier (capped at 0.2) to encourage multi-meta.
- No confidence: small penalty (-0.05).

**Verification (검증)**: Success is measured by ECE and confidence trajectory health on evaluation. Target: ECE < 0.15, healthy rise/drop patterns.

#### 5.1.4 R4: verify_outcome_reward (weight 0.2)

**Intent (의도)**: Reward independent verification that leads to correct answers, and mildly penalize verification that accompanies wrong answers (since wrong-answer verification is at best wasted tokens and at worst false reassurance).

**Hypothesis (가설)**: By conditioning the verify reward on outcome (correct: +0.2, wrong: -0.1), the model will learn to verify effectively rather than ritualistically.

**Implementation** (`rewards.py`, line 1483):

1. Search the post-meta region for verification patterns: "substitute back," "plug in," "sanity check," "verify by," "boundary," "special case," etc.
2. If verification pattern found AND correct: +0.2.
3. If verification pattern found AND wrong: -0.1.
4. If no verification pattern: 0.0.

**Verification (검증)**: Success is measured by verify_effectiveness (proportion of verify events that catch errors). Target: > 0%.

## 6. veRL Migration Plan

### 6.1 Current Architecture

The current training pipeline uses TRL (v0.19.1) with a GDPO monkey-patch:

```
TRL GRPOTrainer
  └── GDPO monkey-patch (_apply_gdpo_patch)
        └── Per-reward group normalization
  └── HF generate (no vLLM)
        └── 4x A100 80GB, Full FT, gradient checkpointing
```

Generation speed is the primary bottleneck. TRL's HF generate produces 16 completions per prompt sequentially on the training GPUs, blocking parameter updates. With 8K-token completions at ~200 tokens/second per GPU, a single GRPO step takes several minutes.

### 6.2 Target Architecture

The migration target is veRL (Volcano Engine Reinforcement Learning), which natively separates rollout generation from training:

```
veRL
  └── vLLM rollout workers (tensor-parallel generation)
  └── FSDP training workers (parameter updates)
  └── GDPO extension (to be ported)
```

An initial vLLM rollout module already exists at `src/rollout/vllm_rollout.py`, which implements 4-GPU tensor-parallel generation for math problems. This module handles prompt formatting, answer extraction, and WandB logging.

### 6.3 Expected Benefits

| Aspect | Current (TRL) | Target (veRL) |
|---|---|---|
| Generation | HF generate, sequential on training GPUs | vLLM, pipelined on dedicated workers |
| Training | Full FT via HF Trainer | FSDP via veRL workers |
| Throughput | ~200 tokens/s/GPU | ~600-1000 tokens/s/GPU (vLLM) |
| Estimated speedup | Baseline | 3-5x generation speed |
| GDPO | Monkey-patch on GRPOTrainer | Native extension module |

### 6.4 Migration Risks

1. **GDPO porting**: The per-reward normalization logic must be re-implemented within veRL's advantage computation. The current monkey-patch intercepts `_generate_and_score_completions`; veRL's architecture may require a different hook point.
2. **Reward function compatibility**: veRL expects reward functions in a different format than TRL's `List[Callable]`. Each reward function must be adapted.
3. **Checkpoint compatibility**: Full FT checkpoints from TRL must be loadable by veRL's FSDP sharding.

Migration is planned as a post-Phase-3 activity and does not block the current E19/E20 experimental sequence.

### 8.3 Historical Context: V5 Seven Findings

For readers unfamiliar with the V5 study (`study_metacot_v5_2026_04_05.md`), the seven core findings were:

1. **All Meta-CoT models underperform base_sft** (-6.4pp to -13.2pp).
2. **Format bug systematically penalized SFT models** (+2.9 to +4.7pp recovery after fix).
3. **Selection bias**: models emit meta on hard problems, creating a -15pp to -38pp confound.
4. **Diagnosis is perfect (0 wrong diagnoses in 30 AIME problems) but execution is absent** (0/20 correct diagnoses led to route switching).
5. **Route switching is necessary and sufficient** for AIME success (3/3 = 100% when switching occurs, 0/27 when it does not).
6. **Token overhead** from meta blocks causes truncation on hard problems (15-25% of tokens consumed by meta).
7. **RL rewards incentivize wrong behaviors**: more reward components correlate with lower accuracy (E10 at 53.9% vs E9c at 60.7%).

The V5 conclusion: "Meta-CoT is a doctor who writes a perfect diagnosis but never writes a prescription."
