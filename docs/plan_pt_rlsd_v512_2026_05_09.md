### 3.3 Worker-side top-K 메서드 (CRITICAL 1 fix)

**v5.11 결함**: `trainer._ref_policy_wg.actor_module` (잘못된 이름 + driver 접근 불가).

`verl/workers/fsdp_workers.py` `ActorRolloutRefWorker` class에 method 추가:
```python
@register(dispatch_mode=Dispatch.DP_COMPUTE_PROTO)
def compute_position_top_k_hits(self, data: DataProto):
    """Worker-side top-K check. Returns [B] bool: META_START in top-K of last valid pos."""
    input_ids = data.batch["input_ids"]
    attn = data.batch["attention_mask"]
    top_k = data.meta_info["top_k"]
    target_id = data.meta_info["target_id"]

    with torch.no_grad():
        out = self.ref_module_fsdp(input_ids=input_ids, attention_mask=attn)
        logits = out.logits  # [B, T, V] — FSDP all-gather 자동

    # Pure tensor logic — extract last valid position
    last_pos = (attn.sum(dim=1).long() - 1).clamp(min=0)
    B = input_ids.size(0)
    last_logits = logits[torch.arange(B, device=logits.device), last_pos]  # [B, V]

    top_k_idx = last_logits.topk(top_k, dim=-1).indices  # [B, K]
    in_top_k = (top_k_idx == target_id).any(dim=-1)  # [B] bool

    return DataProto.from_dict(tensors={"top_k_hits": in_top_k.cpu()})
```

driver-side caller (`verl_sdc.py`):

## 5. OPD-Decoy 결정 (CONCERN 5 fix — 정당화 정확성)

**v5.11 결함**: "forking 비용 크다"는 모호.

**v5.12 fix**: 명확한 cost 분석.

veRL에 OPD top-K KL 추가하려면:
- Generic `actor.aux_loss_hook` 같은 extension point는 **존재하지 않음** (확인됨)
- 가능한 방법: `verl/workers/actor/dp_actor.py:compute_policy_loss` monkey-patch (verl_sdc.py가 이미 같은 패턴으로 `ray_trainer.compute_advantage` patch 중)
- **진짜 cost**: per-token KL은 ref logits을 학습 step마다 cache 또는 매번 recompute. R5는 sequence-aggregate scalar (logp_T - logp_S만 저장)이고, OPD-Decoy는 [B, T, K=64] full distribution 저장 → 메모리 +25-40% (8B model H200 4×141GB에서 OOM 가능)
- **결론**: TRL 유지. memory cost가 실제 prohibitive factor (forking 자체 cost는 patchable).

OPD R8 (TRL, H200 141GB) 정상 학습 신호 확인됨 (step 1 reward 0.32, kl_pos 2.24, kl_neg 2.26). preempt + push 인프라 fix 적용됨.
